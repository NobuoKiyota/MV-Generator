# Antigravity プロジェクト開発ルール

## 1. 動作確認（実機検証）の徹底
- ソースコードの記述完了（文法上の問題がないこと）のみで「完了」と報告することを**厳禁**とします。
- 実装や修正を行った後は、必ずローカル環境（実機）においてアプリケーションを起動し、テストの実行や画面表示、動作検証プロセスを経て、実際に意図通りに動作することを確認しなければなりません。
- 「完了」を報告する際は、必ずどのような検証を行い、どのような結果が得られたか（動作ログや出力確認など）の具体的な証拠を含めて報告してください。
- **🚨 虚偽報告・不十分な検証の厳禁 (極めて重要)**:
  - Tkinter などの GUI アプリケーションを修正した際は、インポート確認だけでなく、必ず操作シナリオをプログラムでシミュレートする検証スクリプト（`verify_gui_scenario.py` 等）を実機上で実行しなければなりません。
  - 完了報告のメッセージには、テストスクリプトが 100% SUCCESS/PASSED となったコマンド実行ログ（stdout/stderr）を**証拠（エビデンス）として必ず添付**してください。エビデンスのない「確認しました」という虚偽の報告は固く禁止します。


## 2. マイルストーン・タスクの細分化
- ユーザーからの依頼や処理内容が大きい場合は、タスクを細かいステップ（マイルストーン）に分割して段階的に実行します。
- 各ステップごとに確実に動作することを確認した上で次のステップに進み、進捗を確実に担保する運用を行います。

## 3. 環境構築・パッケージ追加プロセスの分離（透明化の徹底）
- パッケージのインストール（`pip install` 等）や仮想環境の変更・構築といった「環境構築系プロセス」は、原則としてAIがバックグラウンドで隠れて実行することを禁止します。
- AIが必要なライブラリや設定の追加を行う際は、`requirements.txt` や設定ファイルを修正した上で、ユーザーに「どのようなパッケージを追加したか」および「実行すべきコマンド（例: `run_xlsx_generator.bat` の実行など）」を提示し、ユーザー自身の操作（あるいはユーザーの監視下にある直接のバッチウィンドウ）にて明示的にインストールと起動を行ってもらいます。これにより、開発プロセスの透明化と Quota の無駄な消費抑制を両立します。
