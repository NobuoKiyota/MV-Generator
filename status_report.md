# 📊 プロジェクト進捗・現状報告書 (Status Report) - 2026/07/01

楽曲の音源（mp3）や歌詞から演出タイムラインを自動設計し、既存MVジェネレーター互換の Excel (.xlsx) を出力する「簡易初稿作成ツール（デスクトップGUI版）」の構築結果、および実装上の反省点のまとめです。

## 📌 1. 今回実装した成果物 (xlsx_generator_gui)

1. **[xlsx_generator_gui.py](file:///z:/mv-generator/xlsx_generator_gui.py)** (新規)
   - Python標準の `tkinter` による軽量なデスクトップGUIアプリ。
   - Webサーバーやポート衝突を気にせず、Windows上でダブルクリックするだけで瞬時に立ち上がる極小フォーム構成。
   - BGM音源、歌詞、BPM、1小節目開始時間（オフセット）の最小入力に加え、新たに **MIDIファイル (.mid)** の読み込みに対応。
2. **[utils.py](file:///z:/mv-generator/utils.py)** (拡張)
   - **MIDI歌詞アライメント同期エンジン (`align_lyrics_with_midi`)**:
     - `mido` ライブラリを使用してMIDI内のテンポマップ（テンポ変更イベント）とメロディ音符（ノートオン/オフ）を解析し、歌詞のフレーズ（行）とミリ秒単位で完全に自動同期した高精度なタイムライン枠（秒数）を計算。
     - Gemini API に対しては「時間計算」を任せず、この確定したタイムライン枠を渡して「演出説明」と「画像プロンプト（英語）」を充填させるだけのハイブリッド設計に変更。
3. **[run_xlsx_generator.bat](file:///z:/mv-generator/run_xlsx_generator.bat) & [launcher.py](file:///z:/mv-generator/launcher.py)** (修正)
   - Python 仮想環境 `venv` への依存パッケージ（新たに `mido` を追加）の自動インストールチェックを行い、デスクトップGUIを直接起動する安全な起動システム。文字コードの競合による即時クラッシュを解消。

## 🧠 2. 実装および対話プロセスにおける反省点

今回の開発プロセスにおける最大の反省点は、**「AI側の都合や過信による、ユーザーのユースケース・技術限界の読み違え」** でした。

### ① ユーザーの本来のユースケース（軽量GUI）の読み違え
- **反省点**: 既存のMVジェネレーター（`app.py`）が Streamlit（Webアプリ）だったため、同じ技術を使った方が手戻りが少ないという開発側（AI）の都合だけで `xlsx_generator_app.py`（Streamlit版）を作ってしまいました。結果、「ただ初稿をパッとExcel出力したいだけなのに、裏でWebサーバーが動きブラウザが立ち上がる大げさな仕組み」を押し付け、ポート競合やバッチのハングアップなど余計なトラブルを発生させてしまいました。
- **今後の対策**: 技術選定の前に、「ダブルクリックでポツンとウィンドウが立ち上がるだけのデスクトップネイティブGUIで動くべきか」「ブラウザを使うべきか」といった最終的な使い勝手のイメージを早期に確認する。

### ② 時間の定量化（アライメント）におけるAIへの過度な依存（無理筋）
- **反省点**: Gemini 2.5 Flash に mp3 データを渡したからといって、時間の正確なタイムコード（ミリ秒単位）の計算や、指定されたBPM小節グリッドへの正確な吸着（スナップ）をプロンプト指示（LLMの脳内計算）だけで行わせようとしたことです。結果として、実際の歌い出し（11.364秒）とAIの出力（6.3秒）で大きなギャップを生み、「適当に書いているように見えて使い物にならない」状態に陥らせてしまいました。
- **今後の対策**: 数値計算や物理的なタイミング同期（アライメント）はLLMが最も苦手とする分野です。これらはローカルの Python コード（今回のMIDIノートオン解析や算術計算）で100%厳密に枠を組み立て、LLMには「その枠を埋めるクリエイティブな演出・プロンプトの提案」のみに専念させるハイブリッド設計を大前提とします。

### ③ ランチャー（launcher.py）の依存性チェック追加漏れ
- **反省点**: `requirements.txt` に新規パッケージ（`mido`）を追加した際、`launcher.py` 内の dependency check code にインポート文を追記し忘れたため、ランチャーが機能せず ModuleNotFoundError でクラッシュさせてしまいました。
- **今後の対策**: 新規ライブラリの導入時は、「requirements.txtの更新」「ランチャーのチェックコードへの追記」「テストコードの検証」を一つのアトミックな作業として徹底する。
